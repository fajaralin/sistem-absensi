Deskripsi & Analisis Sistem

Definisi & Aliran Bisnis: Use Case Diagram menggambarkan interaksi batas sistem (system boundary) antara aktor (entitas eksternal) dengan fungsionalitas presensi biometrik pada lobi sekolah. Terdapat tiga aktor utama yang diidentifikasi dalam sistem: Siswa sebagai pelaku utama kegiatan belajar mengajar yang mengisi presensi, Administrator sebagai pengelola data master dan sistem, serta Python AI Engine sebagai aktor sistem terkomputerisasi eksternal yang diintegrasikan untuk menangani pengenalan wajah secara real-time. Use case utama bagi Siswa mencakup "Melakukan Presensi Wajah" dan "Mengajukan Sakit/Izin" secara mandiri di kiosk. Pengajuan sakit/izin memiliki hubungan include dengan "Mengunggah Surat Bukti", yang menegaskan bahwa secara prosedural sistem mewajibkan pengunggahan lampiran berkas pendukung medis atau surat persetujuan orang tua sebagai bukti sah ketidakhadiran siswa.

Implementasi Kode Program: Dari hasil rekayasa balik struktur kode aplikasi, use case "Melakukan Presensi Wajah" dipetakan langsung pada halaman depan kiosk yang dikendalikan oleh berkas template absensi.blade.php dan diproses pada controller PublicAbsensiController.php melalui method scanAuto(). Use case "Mengajukan Sakit/Izin" diproses melalui form input di sisi client dan ditransmisikan ke server menggunakan method submitIzin() pada controller yang sama, yang kemudian mendelegasikan logika penyimpanannya ke dalam database melalui class service AttendanceService.php di method submitStudentLeave(). Di sisi admin, pengelolaan data master, penyesuaian parameter radius koordinat GPS, dan verifikasi log kehadiran dipetakan pada folder admin controller, khususnya pada berkas SettingController.php.

Rekayasa Balik & Pola Desain: Melalui analisis rekayasa balik, use case ini menunjukkan pemisahan hak akses (Role-Based Access Control) yang sangat ketat untuk mengamankan data kehadiran sekolah. Dengan membatasi akses siswa hanya pada fungsi presensi wajah lobi dan pengisian izin, sistem meminimalisir celah eksploitasi data kehadiran. Relasi dependensi include pada use case unggah dokumen membuktikan bahwa sistem menerapkan validasi berlapis di mana model database Attendance dirancang untuk menerima field face_image_path opsional yang dialokasikan khusus untuk menyimpan path file dokumen surat izin dalam folder lokal storage/app/public/leave_attachments.

Deskripsi & Analisis Sistem

Definisi & Aliran Bisnis: Activity Diagram ini memetakan alur perilaku dinamis dari interaksi siswa di depan mesin kiosk presensi lobi sekolah. Alur utama berfokus pada validasi lokasi di sisi client dan inisialisasi hitung mundur 3 detik sebelum pemindaian biometrik otomatis untuk presensi dilakukan. Desain alur aktivitas ini membagi skenario siswa: jika siswa terdeteksi di dalam radius sekolah, hitung mundur berjalan untuk melakukan presensi. Sebaliknya, jika siswa di luar radius (seperti saat membuka web dari rumah), auto-scan presensi dikunci (nonaktif), namun sistem tetap membuka akses mandiri agar siswa dapat mengklik tombol "Ajukan Sakit/Izin". Siswa kemudian memindai wajahnya untuk identifikasi profil, mengisi data alasan ketidakhadiran, mengunggah surat, dan menyimpannya.

Implementasi Kode Program: Secara implementasi, alur validasi lokasi sisi client ditangani menggunakan HTML5 Geolocation API di dalam method fetchLocation() pada file blade kiosk. Jarak dihitung menggunakan formula Haversine di method calculateDistanceClient(). Pengaturan hitung mundur 3 detik dikelola di Alpine.js oleh method startAutoScan(), yang akan menahan pemindaian otomatis selama 3 detik melalui setInterval. Jika tombol pengajuan izin diklik, sistem memicu method switchToLeaveMode() yang menonaktifkan mode presensi secara instan, menghentikan loop deteksi kehadiran, dan mengaktifkan kamera kembali untuk memverifikasi wajah pemohon izin tanpa mencatat check-in.

Rekayasa Balik & Pola Desain: Rekayasa balik aliran ini mengidentifikasi pola penanganan kesalahan dan asinkronus (Asynchronous State Management). Sistem menggunakan variabel reaktif Alpine.js seperti step, isLeaveMode, analyzing, dan isScanning untuk mengontrol visibilitas elemen UI. Fitur countdown 3 detik dan validasi scanMode di method silentScan() berfungsi sebagai tameng pelindung (*guard*) untuk memecahkan bug race condition. Tanpa delay dan proteksi mode ini, server akan memproses data scan kehadiran siswa secara otomatis (karena request dikirim di background sebelum UI sempat berganti), yang mengakibatkan siswa yang ingin izin malah tidak sengaja tercatat hadir terlambat.

Deskripsi & Analisis Sistem

Definisi & Aliran Bisnis: Sequence Diagram memodelkan interaksi pertukaran pesan (message passing) yang teratur antar elemen sistem berdasarkan urutan kronologis waktu. Kejadian dipicu ketika siswa berdiri di depan kamera kiosk. Setelah hitung mundur 3 detik selesai di sisi client, kamera menangkap frame gambar, mengubahnya menjadi format Base64, dan mengirimkannya ke endpoint controller Laravel. Request tersebut disertai parameter koordinat latitude, longitude, dan tujuan pengiriman (*purpose*). Backend Laravel memvalidasi aturan waktu gerbang dan koordinat GPS. Jika semua validasi lolos, Laravel mendelegasikan pemrosesan pencocokan wajah biometrik ke server Flask Python pada port 5001. Python mendeteksi kecocokan wajah, dan jika terverifikasi, database MySQL akan merekam kehadiran siswa tersebut.

Implementasi Kode Program: Urutan pesan di atas diwujudkan dalam pemanggilan fungsi lintas teknologi. Dari sisi client, pengiriman data base64 diproses menggunakan fungsi fetch('/absensi/scan-auto'). Request ini ditangkap oleh route post Laravel yang mengarah ke method scanAuto() di PublicAbsensiController. Logika validasi dan orkestrasi diatur sepenuhnya di dalam AttendanceService di method scanFacePresenceAuto(). Komunikasi keluar menuju server Python didelegasikan pada service class FaceRecognitionService yang memicu curl request HTTP POST ke endpoint Python API /identify. Hasil olahan data disimpan ke database melalui repository class yang mengimplementasikan query Eloquent ORM.

Rekayasa Balik & Pola Desain: Dari kacamata rekayasa balik, diagram ini memperlihatkan pola integrasi arsitektur terdistribusi (*Decoupled Integration*). Laravel bertindak sebagai gerbang terpusat yang melindungi Python AI Engine dari akses langsung client. Semua request wajib melalui otentikasi token CSRF Laravel dan validasi Geolocation terlebih dahulu sebelum dikirim ke port 5001. Jika validasi lokasi gagal (misalnya jarak melebihi radius sekolah), backend Laravel akan langsung mengembalikan respon error 422 tanpa memicu API Python. Keputusan desain ini sangat penting untuk menghemat resource komputasi CPU server dari ancaman overload akibat proses identifikasi biometrik beruntun yang tidak valid.

Deskripsi & Analisis Sistem

Definisi & Aliran Bisnis: Class Diagram menggambarkan struktur statis dan hubungan asosiasi logis antar komponen berorientasi objek dalam kode sistem presensi. Arsitektur aplikasi dirancang mengikuti pola pemisahan lapisan (Separation of Layers) yang bersih untuk menyederhanakan siklus pemeliharaan aplikasi. Pengontrol web (`PublicAbsensiController`) tidak memproses logika bisnis kehadiran secara langsung, melainkan bergantung pada kelas service (`AttendanceService` dan `SettingService`). Logika bisnis ini kemudian berinteraksi dengan database melalui abstraksi penyimpanan data (interface Repository) untuk memisahkan logika kueri data dari sistem manajemen database fisik (DBMS).

Implementasi Kode Program: Dependensi antar kelas ini diwujudkan dengan memanfaatkan fitur **Dependency Injection** pada Laravel Service Container. Dalam berkas `PublicAbsensiController.php`, Anda dapat mengamati bahwa kelas repositori seperti StudentRepositoryInterface dan kelas service diinjeksikan secara otomatis melalui parameter constructor. Kelas `FaceRecognitionService` mengimplementasikan interface `FaceRecognitionServiceInterface`, bertindak sebagai adapter untuk memanggil subproses API Python Flask. Repository kelas seperti `StudentRepository` dan `AttendanceRepository` mempergunakan Eloquent database model (seperti model `Student`, `User`, `Attendance`, `Setting`) untuk mengoperasikan data ke tabel MySQL secara aman.

Rekayasa Balik & Pola Desain: Rekayasa balik ini menunjukkan kepatuhan sistem yang luar biasa terhadap prinsip **SOLID**, khususnya *Single Responsibility Principle* (SRP) dan *Dependency Inversion Principle* (DIP). Dengan merancang dependensi menggunakan interface (seperti `StudentRepositoryInterface` dan `FaceRecognitionServiceInterface`), komponen tingkat tinggi tidak terikat langsung dengan detail teknis tingkat rendah. Sebagai contoh, jika tim pengembang berniat mengganti mesin AI lokal dengan sistem pengenalan wajah cloud (seperti AWS Rekognition), mereka hanya perlu menulis kelas baru yang mengimplementasikan `FaceRecognitionServiceInterface` tanpa harus merombak satu baris pun kode di dalam `AttendanceService` atau controller.

Deskripsi & Analisis Sistem

Definisi & Aliran Bisnis: Package Diagram mengilustrasikan pengelompokan logis dari berkas-berkas program ke dalam subsistem atau paket terpisah. Pembagian paket ini bertujuan untuk membatasi ketergantungan antar-modul (*decoupling*) sehingga struktur aplikasi menjadi lebih modular dan teratur. Paket dibagi menjadi beberapa lapisan vertikal: Presentation Layer yang menangani tampilan visual dan request client, Business Service Layer sebagai pusat aturan bisnis aplikasi, Infrastructure Layer yang melayani integrasi eksternal, Repository Layer untuk enkapsulasi interaksi basis data, Domain Model Layer yang merepresentasikan skema tabel data, serta External Python AI Engine sebagai modul pemroses algoritma kecerdasan buatan.

Implementasi Kode Program: Pada struktur berkas Laravel, pengemasan ini terpetakan secara konsisten. Paket `PresentationLayer` diwakili oleh berkas controller di folder `app/Http/Controllers` dan file HTML/Blade di `resources/views`. Paket `BusinessService` dan `InfrastructureService` terstruktur di dalam direktori `app/Services` yang terbagi atas file interface dan berkas implementasi konkret. `RepositoryLayer` terstruktur di dalam direktori `app/Repositories/Contracts` and `app/Repositories/Eloquent` untuk implementasi kelas. Paket `ExternalPythonAIEngine` terisolasi sebagai folder eksternal di tingkat root project, menampung file `python_server.py` dan aset berkas XML/JSON dataset wajah siswa.

Rekayasa Balik & Pola Desain: Rekayasa balik pada pembagian kemasan ini membuktikan penerapan aturan dependensi satu arah (One-Way Dependency Rule). Lapisan atas diperbolehkan mengakses lapisan yang berada di bawahnya, namun lapisan bawah dilarang keras untuk mengimpor atau memanggil fungsionalitas dari lapisan di atasnya. Komunikasi keluar menuju `ExternalPythonAIEngine` dijembatani secara eksklusif oleh adapter di `InfrastructureServiceLayer`, sehingga web server Laravel dapat memperlakukan sistem AI Python sebagai *black box*. Desain ini mempermudah perawatan kode, di mana perubahan internal pada dataset AI Python tidak akan mempengaruhi kestabilan kode PHP di Presentation Layer.

Deskripsi & Analisis Sistem

Definisi & Aliran Bisnis: Infrastruktur Diagram (Deployment Diagram) memetakan arsitektur fisik dari server komputasi, perangkat periferal, dan saluran komunikasi yang mendistribusikan sistem presensi biometrik. Topologi ini terdiri atas empat node utama: Client Node yang dioperasikan pada komputer tablet/PC kiosk lobi yang terhubung dengan webcam fisik dan sensor GPS browser, Laravel Web Server Node yang menjalankan engine PHP-FPM untuk pemrosesan MVC dan logika data, Database Node yang mengelola data relasional melalui DBMS MySQL, serta Python AI Node sebagai server pengenal biometrik terdedikasi yang mendengarkan request pada port 5001 lokal menggunakan protokol TCP/IP.

Implementasi Kode Program: Dalam koding aplikasi Laravel, hubungan komunikasi antar node ini dikonfigurasi melalui berkas variabel lingkungan `.env`. Parameter `DB_HOST`, `DB_PORT`, `DB_DATABASE`, `DB_USERNAME`, dan `DB_PASSWORD` mengatur konektivitas langsung ke Database Node. Alamat Python AI Node didefinisikan secara asinkron di file `config/services.php` atau file `.env` menuju port `5001`. Proses otentikasi biometrik dilakukan dengan mengirimkan request HTTP klien dari Laravel Web Server Node ke Python AI Node menggunakan pustaka PHP cURL atau HTTP Client Laravel. Laravel juga memiliki kendali memicu CLI Windows menggunakan instruksi proc_open untuk meluncurkan `python_server.py` secara otomatis jika port 5001 terdeteksi belum aktif.

Rekayasa Balik & Pola Desain: Melalui rekayasa balik infrastruktur terdistribusi ini, keputusan desain pemisahan server web Laravel dan server AI Python terbukti sangat vital untuk menjaga stabilitas kiosk. Pustaka komputasi AI biometrik wajah (seperti OpenCV, Face Recognition) membutuhkan resource komputasi memori RAM dan siklus clock CPU yang tinggi selama memproses algoritma pencocokan matriks citra wajah LBPH. Dengan memisahkan beban pemrosesan ini ke server port 5001 (Python Node), Laravel Server Node dapat fokus secara optimal menangani ribuan transaksi request presensi dari client dengan cepat, tanpa harus terhambat oleh beban proses komputasi citra wajah yang intensif.
